## Tasks

### 4.1 — REQ-009: Fix `engagement_by_day` weekday calculation

**Problem**: `generate_engagement_by_day()` in `chartGenerator.js:241-258` uses `hour % 7` to derive the weekday index. This is mathematically wrong — the `audience_activity_hour` field is 0-23 (hour of day), so `hour % 7` produces a rotating pattern that has nothing to do with the actual day of the week.

**Current (broken)**:
```js
// chartGenerator.js:247-249
const hour = parseInt(item.audience_activity_hour);
const dayIndex = hour % 7;
```

**Fix**: The Windsor `audience_activity` data includes a `date` field (format `YYYY-MM-DD`). Use `new Date(item.date).getDay()` to get the real weekday (0=Sunday). Remap to Monday-first since `dayNames` starts with Hétfő.

**Implementation**:
```js
this.activity.forEach(item => {
    const count = parseInt(item.audience_activity_count) || 0;
    const jsDay = new Date(item.date).getDay(); // 0=Sun, 1=Mon, ..., 6=Sat
    const dayIndex = jsDay === 0 ? 6 : jsDay - 1; // 0=Mon, ..., 6=Sun
    dayData[dayIndex] += count;
    dayCounts[dayIndex]++;
});
```

**Files**:
- `backend/services/chartGenerator.js` — `generate_engagement_by_day()` (line ~241)

**Verify**: After fix, the chart should show higher activity on weekdays (typical for TikTok business accounts). If all days show roughly equal values, the fix works correctly — previous `hour % 7` would have clustered hours 0-6 into the first 7 "days" artificially.

### 4.2 — REQ-010: Facebook post/reel deduplication by `post_id`

**Problem**: TikTok videos have `_dedupeVideos()` (line ~285) that deduplicates by `video_id`. Facebook posts and reels have NO equivalent deduplication. Windsor returns one row per `(date, post_id)` combination, so multi-day queries return duplicate posts with different daily metrics. This inflates counts in `fb_all_posts`, `fb_top_3_posts`, and `generateFacebookReelTable`.

**Implementation**: Create `_dedupeFacebookPosts(posts)` mirroring the TikTok pattern:

1. Group by `post_id`
2. For each group: take max of `post_impressions`, `post_reach`, `post_reactions`, `post_comments`, `post_shares`, `post_clicks`, `post_video_views*`
3. Keep first row's `post_message`, `post_created_time`, `post_permalink`

Apply dedup in:
- `generate_fb_all_posts()` — currently raw `this.video`
- `generate_fb_top_3_posts()` — currently sorts raw `this.video`
- `generateFacebookReelTable()` — filters raw `posts` param

**Files**:
- `backend/services/chartGenerator.js` — new `_dedupeFacebookPosts()` method + update 3 callers

### 4.3 — Remove dead `/api/evaluations/:id/react` backend endpoint

**Status: CANCELLED — endpoint is NOT dead.**

Investigation shows the backend endpoint at `server.js:1451` (`app.patch('/api/evaluations/:id/react', ...)`) is duplicated by a frontend Next.js API route at `frontend/src/app/api/evaluations/[id]/react/route.ts` that calls Prisma directly. The frontend route bypasses the Express backend entirely.

**Decision**: The backend `/react` endpoint is dead *from the frontend's perspective* (no frontend code calls it via the Express proxy). However, removing it is a separate cleanup concern, not a data quality fix.

**Action**: Move to Phase 5 cleanup list. Remove `server.js:1450-1478` (the `PATCH /api/evaluations/:id/react` handler) since the frontend handles this with its own API route + Prisma. Also verify no external caller depends on the Express endpoint (check for any mobile app or webhook usage — currently none).
