## Task 1: Fix Race Condition in Messages JSON Append (REQ-005)

### Problem

Three endpoints do read-modify-write on `Evaluation.messages` (Json field):

1. **POST `/api/evaluations`** (route.ts:71-85) — admin sends message
   - Reads existing messages (line 72-75), pushes new msg (line 77), writes back (line 85)
2. **PATCH `/api/evaluations/[id]/reply`** (reply/route.ts:49-59) — client replies
   - Reads evaluation (line 25-28), pushes to messages (line 50), writes back (line 52-59)
3. **PATCH `/api/evaluations/[id]`** (route.ts:40-48) — reaction update
   - Reads evaluation (line 17), mutates reaction field in array (line 43-44), writes back (line 47)

If admin sends a message while client is replying, both read the same messages array, each appends their own message, and the second write overwrites the first — **one message is permanently lost**.

### Root Cause

Classic TOCTOU (time-of-check-time-of-use): the read and write are separate Prisma operations with no transaction isolation.

### Solution: Atomic JSON Append via `$executeRaw`

Replace read-modify-write with a single PostgreSQL `jsonb_concat` (`||`) operation. PostgreSQL handles atomicity at the row level.

**Target SQL pattern (append):**
```sql
UPDATE "Evaluation"
SET "messages" = COALESCE("messages", '[]'::jsonb) || $1::jsonb,
    "updatedAt" = NOW()
WHERE "id" = $2
```

**Target SQL pattern (reaction update on specific index):**
```sql
UPDATE "Evaluation"
SET "messages" = jsonb_set(
  COALESCE("messages", '[]'::jsonb),
  ARRAY[$1::text],  -- index as text
  jsonb_set(("messages"->$1::int)::jsonb, '{reaction}', $2::jsonb)
),
    "updatedAt" = NOW()
WHERE "id" = $3
```

### Implementation Steps

#### Step 1.1: Create helper function `appendEvaluationMessage`

File: `frontend/src/lib/evaluation-db.ts` (new)

```typescript
import { prisma } from '@/lib/prisma';
import { Prisma } from '@prisma/client';

interface ChatMessage {
  role: 'admin' | 'client';
  text: string;
  at: string;
  name?: string;
}

/**
 * Atomically append a message to Evaluation.messages JSON array.
 * Uses PostgreSQL jsonb concatenation — no read-modify-write.
 */
export async function appendEvaluationMessage(
  evaluationId: string,
  message: ChatMessage
): Promise<void> {
  const msgJson = JSON.stringify([message]);
  await prisma.$executeRaw`
    UPDATE "Evaluation"
    SET "messages" = COALESCE("messages", '[]'::jsonb) || ${msgJson}::jsonb,
        "updatedAt" = NOW()
    WHERE "id" = ${evaluationId}
  `;
}

/**
 * Atomically set a reaction on a specific message by index.
 * Returns updated row count (0 = not found or index out of bounds).
 */
export async function setMessageReaction(
  evaluationId: string,
  messageIndex: number,
  reaction: string | null
): Promise<number> {
  const reactionJson = reaction ? JSON.stringify(reaction) : 'null';
  const idxText = messageIndex.toString();
  return prisma.$executeRaw`
    UPDATE "Evaluation"
    SET "messages" = jsonb_set(
      COALESCE("messages", '[]'::jsonb),
      ARRAY[${idxText}, 'reaction'],
      ${reactionJson}::jsonb
    ),
        "updatedAt" = NOW()
    WHERE "id" = ${evaluationId}
    AND jsonb_array_length(COALESCE("messages", '[]'::jsonb)) > ${messageIndex}
  `;
}
```

- Send `{ messageIndex, reaction }` instead of full messages array

```typescript
const handleMessageReaction = async (globalIdx: number, emoji: string) => {
  const targetMsg = chatMessages[globalIdx];
  if (!targetMsg) return;
  setEmojiPickerIdx(null);

  for (const ev of evaluations) {
    const msgs = (ev.messages as ChatMessage[] || []);
    const localIdx = msgs.findIndex(
      m => m.text === targetMsg.text && m.at === targetMsg.at && m.role === targetMsg.role
    );
    if (localIdx === -1) continue;

    const newReaction = msgs[localIdx].reaction === emoji ? null : emoji;

    // Optimistic update
    const updated = [...msgs];
    updated[localIdx] = { ...updated[localIdx], reaction: newReaction ?? undefined };
    setEvaluations(prev => prev.map(e => e.id === ev.id ? { ...e, messages: updated } : e));

    // Atomic server update
    try {
      await fetch(`/api/evaluations/${ev.id}`, {
        method: 'PATCH',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ messageIndex: localIdx, reaction: newReaction }),
      });
    } catch {}
    break;
  }
};
```

#### Step 1.6: Delete dead `/react` endpoint

File: `frontend/src/app/api/evaluations/[id]/react/route.ts` — DELETE entire file.

This endpoint uses the legacy `clientReaction` scalar field and is not called by any current UI code. The `EvaluationBubble` uses per-message reactions via the generic PATCH endpoint.

### Edge Cases to Handle

| Scenario | Handling |
|----------|----------|
| First message (no evaluation row yet) | Upsert creates row, then atomic append |
| Concurrent admin + client message | Both use `||` append — PostgreSQL serializes at row level |
| Reaction on deleted/nonexistent index | `jsonb_array_length` guard returns 0 rows updated |
| Empty/null messages column | `COALESCE("messages", '[]'::jsonb)` handles null |
| Legacy evaluations with no messages array | COALESCE handles this; legacy `adminMessage`/`clientReply` fields untouched |

### Rollback Strategy

If `$executeRaw` causes issues (e.g., Prisma version incompatibility):
- Fallback: wrap read-modify-write in `prisma.$transaction` with `isolationLevel: Serializable`
- This is slower but still prevents lost updates
- No schema changes needed for either approach
