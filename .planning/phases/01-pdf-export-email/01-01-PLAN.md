# Phase 1: PDF Export & Scheduled Email

## Objective

Deliver production-ready server-side PDF reports with brand header/footer, KPI cards, charts (as server-rendered images), daily tables, and top content tables -- properly paginated across A4 pages. Wire the existing cron job to reliably send these PDFs as email attachments on each company's configured `emailDay`/`emailHour`.

## Current State Analysis

### What exists:
- **Client-side PDF** (`frontend/src/lib/exportPdfClient.ts`): `html-to-image` + `jsPDF` DOM capture. Works for on-screen export but cannot run server-side (no DOM).
- **Server-side PDF** (`backend/services/pdfService.js`): Puppeteer renders HTML string to A4 PDF. Already functional.
- **Report HTML template** (`backend/services/reportTemplate.js`): Light-theme HTML with KPI grid + daily table + top videos. No charts (images).
- **Dashboard PDF template** (`backend/services/dashboardPdfTemplate.js`): Dark-theme HTML with KPI grid, video table, daily trend tables. No chart images either.
- **Monthly report cron** (`backend/services/monthlyReportJob.js`): Runs hourly via `node-cron`, matches `emailDay`/`emailHour` per company. Uses `reportTemplate.js` (light theme). Already fetches Windsor data live or from `ReportCache`.
- **Email service** (`backend/services/emailService.js`): Resend with attachment support. Working.
- **Prisma schema**: `Company.emailDay` (1-28, default 3), `Company.emailHour` (0-23 UTC, default 8). No `emailEnabled` toggle.

### Key gaps:
1. **No charts in PDF** -- both templates render KPI cards + tables but zero chart images
2. **No brand header/footer with page numbers** -- current templates have a simple header, no repeated footer, no pagination marks
3. **Template mismatch** -- `monthlyReportJob.js` uses the light `reportTemplate.js`, but the dashboard uses the dark `dashboardPdfTemplate.js`. Need one canonical branded template.
4. **No `emailEnabled` toggle** -- no way for a company to opt out of scheduled emails
5. **Cron only runs when `ENABLE_BILLING=true`** -- blocks dev/testing without billing enabled
6. **No chart.js server-side rendering** -- need `chartjs-node-canvas` for generating chart PNGs on the backend
7. **Email body is inline HTML in `monthlyReportJob.js`** (`buildEmailBody`) -- not using the branded `email.ts` templates

### Task 3: Integrate chart rendering into PDF generation pipeline

**File**: `backend/services/pdfService.js`
**Changes**:
- Update `generateReportPdf(html)` to accept optional `headerTemplate` and `footerTemplate` parameters:
```js
async function generateReportPdf(html, options = {}) {
  // ...existing browser launch...
  const pdfBuffer = await page.pdf({
    format: 'A4',
    printBackground: true,
    displayHeaderFooter: !!options.headerTemplate,
    headerTemplate: options.headerTemplate || '',
    footerTemplate: options.footerTemplate || '',
    margin: {
      top: options.headerTemplate ? '35mm' : '15mm',
      right: '12mm',
      bottom: options.footerTemplate ? '25mm' : '15mm',
      left: '12mm',
    },
  });
}
```
- Keep backward compatibility: existing callers without options work unchanged

**File**: `backend/services/reportPipeline.js` (NEW)
**Changes**:
Create an orchestrator function that the cron job and the API route both call:
```js
async function generateFullReport({
  companyName, platform, platformLabel, month,
  kpis, dailyMetrics, topVideos, adminNote,
  chartConfigs,  // Array of Chart.js config objects from chartGenerator.js
}) → Promise<Buffer>
```

1. Render each `chartConfigs[i]` via `chartRenderer.renderChartToDataUrl()`
2. Build HTML via `brandedPdfTemplate.buildBrandedReportHtml()` with the chart data URLs
3. Generate PDF via `pdfService.generateReportPdf()` with header/footer templates
4. Return PDF buffer

## Architecture Decisions

### ADR-001: Server-side chart rendering with chartjs-node-canvas

**Status**: Proposed

**Context**: Current PDFs have KPI cards and tables but no charts. The frontend renders charts with Chart.js in the browser. We need chart images in server-generated PDFs (for email attachments and API export).

**Decision**: Use `chartjs-node-canvas` (wraps Chart.js + node-canvas) to render Chart.js configs to PNG buffers on the backend. Reuse the same chart config objects that `backend/services/chartGenerator.js` already produces.

**Consequences**:
- (+) Reuses existing chart config logic -- no new charting library to learn
- (+) Charts in PDFs match dashboard charts exactly (same Chart.js rendering)
- (+) No browser needed for chart rendering (lighter than Puppeteer screenshots)
- (-) `node-canvas` requires native dependencies (cairo, pango) -- must be in Docker image
- (-) Adds ~50MB to Docker image size for native canvas libs

### ADR-002: Single branded template replaces two existing templates

**Status**: Proposed

**Context**: Two PDF templates exist: `reportTemplate.js` (light, used by cron job) and `dashboardPdfTemplate.js` (dark, used by export API). They diverge in design and capabilities.

**Decision**: Create `brandedPdfTemplate.js` as the single source of truth. It uses the brand design system from `email.ts` (navy/teal palette). Both the cron job and the export API route use this template.

**Consequences**:
- (+) One template to maintain, consistent brand across all PDFs
- (+) Brand header/footer with page numbers on every page
- (-) Deprecates `dashboardPdfTemplate.js` dark theme -- clients who prefer dark PDF lose that option
- (mitigable) Could add a `theme: 'light' | 'dark'` parameter later if needed

### ADR-003: Backend email template module (CJS) mirrors frontend email.ts

**Status**: Proposed

**Context**: The branded email layout lives in `frontend/src/lib/email.ts` (TypeScript, ESM-ish). The backend is CJS. Cross-importing is fragile.

**Decision**: Create `backend/services/emailTemplates.js` with the same BRAND constants and layout helpers, in CJS. Accept the duplication (small surface area, ~50 lines of shared constants).

**Consequences**:
- (+) No cross-module import complexity
- (+) Backend email sends work independently of frontend build
- (-) BRAND constants duplicated -- must update in two places if brand changes
- (mitigable) Extract to a shared `packages/brand/` later if duplication becomes a problem
